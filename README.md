# PSA — Persistent Semantic Atlas

### Typed memory objects, anchor-based retrieval, and a trained selector for AI agents.

<br>

Every conversation you have with an AI — every decision, every debugging session, every architecture debate — disappears when the session ends. Other memory systems extract snippets and throw away the reasoning. PSA takes a different approach: **ingest everything, derive typed memory objects from it, and retrieve by learned anchor regions — not flat embedding search.**

## How It Works

PSA has two pipelines that share a single SQLite-backed memory store.

### Ingest Pipeline

Raw source files → typed memory objects derived by a local model.

```
psa mine <dir>
    │
    ├── structure-aware chunking (fine / mid / section)
    ├── Qwen2.5-7B-Instruct (local via Ollama) → typed MemoryObjects
    ├── retention filter (score ≥ 0.65)
    ├── deduplication (cosine > 0.92 → merge)
    └── embed (BAAI/bge-base-en-v1.5) + persist to SQLite
```

Six memory types: **EPISODIC** (specific events), **SEMANTIC** (facts and concepts), **PROCEDURAL** (how-to), **FAILURE** (what went wrong), **TOOL_USE** (tool/API behavior), **WORKING_DERIVATIVE** (scratch notes).

Raw source records are **immutable** — the derived memory objects are indexes over them, not replacements.

### Query Pipeline

```
psa search "query"
    │
    ├── embed query (bge-base-en-v1.5)
    ├── AnchorRetriever — BM25 + dense, fused via RRF → top-32 anchor candidates
    │     BM25 index includes anchor names, meanings, include_terms, generated
    │     query patterns, and accumulated real-query fingerprints
    ├── AnchorSelector — cosine baseline or trained cross-encoder → 1–4 anchors
    │     Cross-encoder input: (query, anchor_card_with_query_patterns)
    ├── fetch MemoryObjects for selected anchors
    └── AnchorSynthesizer — one LLM call → coherent narrative paragraph (~700 tokens)
          Fallback: EvidencePacker role-organized sections if LLM unavailable
```

Each anchor card includes `generated_query_patterns` — 10–15 example questions seeded at atlas build time that describe what queries the anchor is likely to answer. At inference time, real queries that match an anchor above a relevance threshold are accumulated as `query_fingerprint` entries (capped at 50 per anchor, FIFO), further enriching BM25 retrieval over time.

`AnchorSynthesizer` produces one coherent narrative paragraph (~700 tokens) from the selected anchors' memories, replacing ranked memory bullets. If the LLM is unavailable, it falls back to `EvidencePacker` role-organized sections.

### Atlas

The atlas is a map of your memory space. Instead of searching every memory individually (slow, noisy), PSA clusters your memories into **anchor regions** — groups of related memories that share a topic. Each anchor gets a short semantic description like `staff-assignment-problems` or `livekit-audio-pipeline` generated by Qwen.

When you search, PSA matches your query to a handful of anchors (not thousands of memories), then fetches the memories behind those anchors. The number of anchors scales automatically with your corpus size (~1 anchor per 5 memories, up to 224 learned + 32 novelty).

## Architecture Deep Dive

This section covers every subsystem in detail: how memories are created, how they're found, how the system learns, and how it forgets.

### Memory Objects and Types

Every memory in PSA is a typed `MemoryObject` with a title, body, summary, embedding, quality score, and anchor assignment. Memories are created via `MemoryObject.create(...)` and stored in a SQLite database (WAL mode) at `~/.psa/tenants/{tenant_id}/memory.sqlite3`.

Six memory types, each with different retention and packing behavior:

| Type | What it captures | Packing priority |
|------|-----------------|-----------------|
| **FAILURE** | What went wrong and why | 1st (highest) |
| **PROCEDURAL** | How-to steps, recipes, workflows | 2nd |
| **TOOL_USE** | Tool/API behavior, quirks, flags | 3rd |
| **EPISODIC** | Specific events, debugging sessions | 4th |
| **SEMANTIC** | Facts, concepts, domain knowledge | 5th |
| **WORKING_DERIVATIVE** | Scratch notes, intermediate reasoning | 6th (lowest) |

Failures are packed first because they prevent repeating mistakes. Working derivatives are packed last because they're transient.

### Consolidation: Raw Text to Typed Memories

The consolidation pipeline (`psa/consolidation.py`) converts raw source text into typed memory objects through three stages.

**Stage 1: Hierarchical Chunking.** Source text is split at three granularity levels:

| Level | Token range | Character range | Purpose |
|-------|------------|-----------------|---------|
| Fine | 80-180 | 320-720 | Individual facts, single events |
| Mid | 220-450 | 880-1800 | Paragraphs, related steps |
| Section | 500-1500 | 2000-6000 | Full topics, complete procedures |

Splitting is structure-aware: it recognizes `# ` headings and double-newlines for prose, `class`/`def` boundaries for code, and `Human:`/`User:` turns for conversations. Adjacent chunks overlap by 40 characters to preserve context at boundaries.

**Stage 2: LLM Extraction.** Chunks are sent to the LLM (cloud or local Ollama) with a system prompt that defines the six memory types and their retention guidelines. The LLM returns structured JSON with type, title, body, summary, retention_score (0.0-1.0), and evidence_chunk_ids linking back to source text.

**Stage 3: Filtering and Deduplication.**
- Memories with `retention_score < 0.65` are dropped
- EPISODIC and SEMANTIC memories additionally require body length >= 30 characters (filters low-value chatter)
- PROCEDURAL, FAILURE, and TOOL_USE memories are kept at the threshold (these are high-value types)
- Near-duplicates (cosine similarity > 0.92 against existing memories of the same type) are marked as duplicates rather than stored twice

Raw source records are persisted immutably in a separate `memory_sources` table. Memory objects are derived indexes over them, not replacements.

### Embeddings

All embeddings use **BAAI/bge-base-en-v1.5** via sentence-transformers:
- **768 dimensions**, L2-normalized (unit vectors)
- Cosine similarity = dot product for unit vectors
- Stored as binary blobs in the `memory_objects` table

### Atlas: Mapping the Memory Space

The atlas is built by `AtlasBuilder.build_atlas()` in `psa/atlas.py`. It uses spherical k-means to cluster memories into semantic regions, then generates a description card for each region.

**Fixed V1 sizing:** 224 learned anchors + 32 novelty anchors = 256 total. This is intentionally fixed rather than adaptive.

**Minimum corpus gate:** 200 memories with embeddings required. Below this, `AtlasCorpusTooSmall` is raised and the tenant stays on the baseline path.

**Spherical k-means:**
- k-means++ initialization on L2-normalized embeddings
- Convergence threshold: centroid shift < 1e-4
- Max 200 iterations per seed
- 3 independent seeds for stability checking

**Stability check:** K-means is run with 3 seeds. Cross-seed agreement is measured via adjusted Rand index (ARI). If stability falls below 0.60 (i.e., instability > 0.40), the build is rejected with `AtlasUnstable`. This prevents fragile atlas proposals from replacing a working one.

**Anchor card generation:** For each cluster, the LLM analyzes the top-10 sample memories and generates:
- A kebab-case name (2-4 words, e.g., `graphql-migration-decisions`)
- A 1-2 sentence meaning describing what belongs here
- Include terms (up to 8) and exclude terms (up to 4) for BM25 matching
- Fallback: stub cards are generated if the LLM is unavailable

**Novelty anchors:** The 32 memories with the lowest max-cosine-similarity to any learned centroid are selected as novelty anchor seeds. These are the "farthest out" points in embedding space — memories that don't fit any learned cluster well. During hot assignment, memories with best-learned-anchor similarity below 0.7 (i.e., cosine distance > `NOVELTY_DISTANCE_THRESHOLD` of 0.3) are routed to the nearest novelty anchor instead.

**Anchor matching during rebuild:** When the atlas rebuilds, new clusters are matched to existing anchors by centroid similarity (greedy bipartite, threshold 0.5). Matched anchors keep their ID and stable card text. Only genuinely novel clusters get new IDs. Old anchors with no match are retired. This preserves anchor identity across rebuilds, which matters because the trained selector learns anchor semantics.

### Retrieval: Finding Relevant Anchors

The retriever (`psa/retriever.py`) uses hybrid search to find the most relevant anchors for a query.

**BM25 (sparse):** Bag-of-words TF-IDF over anchor card text (name, meaning, include/exclude terms). Parameters: k1=1.5, b=0.75. Tokenization lowercases, strips punctuation, removes 30 stop words, requires minimum 2-character tokens.

**Dense:** Query embedding dot-product against anchor centroids via FAISS (or numpy fallback).

**Reciprocal Rank Fusion (RRF):** Both rankings are fused with k=60:
```
RRF_score(anchor) = 1/(k + dense_rank) + 1/(k + bm25_rank)
```
This produces a balanced ranking where neither signal dominates. Returns the **top-24 candidates**.

### Selection: Choosing the Best Anchors

The selector (`psa/selector.py`) narrows 24 candidates down to 1-4 anchors.

**Cosine mode (baseline):** Re-ranks candidates by dense similarity score. Always available, no dependencies. Returns up to 4 anchors above threshold (default 0.0 — effectively returns top 4).

**Trained mode (cross-encoder):** A fine-tuned `cross-encoder/ms-marco-MiniLM-L-6-v2` scores each (query, anchor_card) pair. The card text is the **stable** version (name + meaning + terms, no volatile examples) so the model remains valid across atlas rebuilds. Max sequence length: 320 tokens. Threshold: calibrated per model via Youden's J (typically 0.3-0.5). Returns up to 4 anchors above the calibrated threshold.

### Packing: Building the Context Window

The packer (`psa/packer.py`) assembles selected memories into a structured context block that fits within a token budget (default 6000).

**Role-organized sections** are packed in priority order:
1. FAILURE WARNINGS
2. PROCEDURAL GUIDANCE
3. TOOL-USE NOTES
4. RELEVANT EPISODES
5. FACTS & CONCEPTS
6. WORKING NOTES

Within each section, memories are ranked by `0.7 * relevance + 0.3 * quality_score`. Greedy packing adds sections until the budget is exhausted. Deduplication uses MD5 hashes (first 12 chars) to prevent the same memory appearing twice, even if it belongs to multiple selected anchors.

### Forgetting: Controlled Memory Decay

PSA doesn't grow unboundedly. The forgetting system (`psa/forgetting.py`) scores memories for disposability:

```
forgetting_score =
    min(idle_days / 90, 1.0)              # idle pressure (caps at 90 days)
  + min(overflow / target, 1.0)           # anchor crowding pressure
  - min(log(1 + pack_count) / 3.0, 1.0)  # usage protection
  - quality_score                          # quality protection
```

Higher score = more disposable. Key behaviors:
- **24-hour grace period:** Newly mined memories get score -10.0 (strongly protected from immediate pruning)
- **Usage protection:** Memories packed ~20 times reach maximum protection
- **Per-anchor pruning:** When an anchor exceeds its budget (default 100 memories), lowest-scoring memories are archived
- **Global cap:** 50,000 active memories maximum. Phase 1 hard-deletes memories archived > 90 days. Phase 2 archives lowest-scoring active memories if still over cap.

The system tracks two usage signals per memory:
- `select_count` — how many times a memory was fetched for a selected anchor
- `pack_count` — how many times it made it into the final packed context (stronger signal)

### Oracle Labeling: Learning What Works

Oracle labeling (`psa/training/oracle_labeler.py`) evaluates which anchor combinations actually help answer real queries. This is how PSA learns task utility rather than just topical similarity.

#### Labeling modes

`psa benchmark longmemeval oracle-label --mode <mode>` supports three strategies:

| Mode | LLM calls | Runtime (500 q) | When to use |
|------|-----------|-----------------|-------------|
| `fast` *(default)* | None | ~25–30 min | Benchmarks with ground-truth anchor IDs (e.g. LongMemEval). Uses deterministic gold-overlap for SupportCoverage only; skips proxy scoring and TaskSuccess. |
| `local` | Local Ollama | ~30–60 min | Production use without a cloud API key. Full two-stage pipeline via `qwen2.5:7b` (or configured model). |
| `api` | Cloud API | ~20–40 min | Production use with a cloud API key. Full two-stage pipeline via the provider in `~/.psa/llm.json`. |

**The `fast` mode is for benchmarks only** — it requires ground-truth anchor IDs to be present in the results file. For production PSA usage (real user sessions with no ground truth), use `local` or `api`.

#### Full two-stage pipeline (local / api modes)

**Stage 1 (cheap, batched):** For each query, generate candidate anchor sets of varying size (1-anchor through 4-anchor combinations from the retriever's top-24). All candidates are scored in a single batched LLM call with proxy metrics.

**Stage 2 (expensive, selective):** Only the top-3 candidates from Stage 1 are evaluated with a full task-success judgment: "Does this context actually help answer the query?" (0.0-1.0 scale from the LLM).

**Oracle score formula:**
```
OracleScore = 0.45 * SupportCoverage
            + 0.20 * TaskSuccess
            + 0.15 * ProceduralUtility
            - 0.10 * NoisePenalty
            - 0.10 * TokenCost
```

Where:
- **SupportCoverage** — fraction of ground-truth anchors covered by the candidate set
- **TaskSuccess** — LLM judgment of context helpfulness (Stage 2)
- **ProceduralUtility** — fraction of selected anchors containing procedural/failure/tool_use memories
- **NoisePenalty** — fraction of selected anchors NOT in the ground-truth set
- **TokenCost** — `min(packed_tokens / 6000, 1.0)` (penalizes bloated context)

The Stage 1 batching reduces ~23 HTTP calls per query to 1, cutting full labeling time from ~16 hours to ~1 hour for 500 queries on an M4 Mac.

### Training Data Generation

The data generator (`psa/training/data_generator.py`) converts oracle labels into cross-encoder training examples with three types:

| Type | Mix ratio | Description |
|------|-----------|-------------|
| **Synthetic** | 60% | Oracle-winning anchors (label=1) + random non-oracle anchors (label=0), split 50/50 |
| **Hard negative** | 20% | Anchors in the retriever's top-24 that the oracle rejected — the hardest negatives |
| **Adversarial** | 20% | Positive examples with query rewrites (lexical compression, conversational phrasing, distractor insertion) mixed with hard negatives for class balance |

Queries are classified into 5 families based on oracle set size: `single_anchor`, `contrastive` (2 anchors, boundary queries), `compositional` (2 anchors, combination), `bridge` (3+ anchors, chain reasoning), and `experience` (episodic retrieval). Training gates require at least 50 examples per active family.

### Selector Training: Three-Phase Curriculum

The selector trainer (`psa/training/train_selector.py`) fine-tunes `cross-encoder/ms-marco-MiniLM-L-6-v2` in three phases:

**Phase 1 — Warm start:** 1 epoch on positives + easy negatives. Learns basic relevance signals.

**Phase 2 — Hard-negative curriculum:** 1 epoch on hard negatives + matched positives. Learns fine boundaries between oracle and near-oracle anchor sets.

**Phase 3 — Adversarial hardening:** 1 epoch on adversarial rewrites + matched hard negatives (shuffled). Builds robustness to query paraphrasing.

Hyperparameters: lr=2e-5, batch=32, max_seq=320, warmup=10%, loss=BCE + 0.2 * margin loss.

**Threshold calibration:** After training, the optimal decision boundary (tau) is computed via Youden's J statistic on the validation set: sweep tau from 0.05 to 0.95, select the value maximizing `sensitivity + specificity - 1`. Fallback: 0.3 if no validation set. The calibrated threshold is saved to `selector_version.json` and loaded at pipeline startup.

**Training gates:** The selector won't train until: >= 300 oracle labels, shortlist recall@24 >= 0.95, and >= 50 examples per active query family. These gates prevent training on insufficient data.

### Health Monitoring and Rebuild Triggers

The health monitor (`psa/health.py`) decides when the atlas needs rebuilding:

| Metric | Threshold | What it means |
|--------|-----------|--------------|
| **Novelty rate** | > 8% | Too many memories landing in novelty anchors — new semantic territory uncovered |
| **Utilization skew** | > 3.0x | Ratio of max to median learned-anchor occupancy — a few anchors are overloaded |

When either threshold is exceeded and new data has arrived, the lifecycle triggers a rebuild. During rebuild, the system falls back to cosine selector automatically and restores the trained selector afterward.

### The Continuous Learning Loop

Everything ties together in the lifecycle pipeline (`psa/lifecycle.py`):

```
                    ┌─────────────────────────────────┐
                    │         NIGHTLY LIFECYCLE        │
                    └──────────────┬──────────────────-┘
                                   │
            ┌──────────────────────┼──────────────────────┐
            ▼                      ▼                      ▼
     ┌─────────────┐      ┌──────────────┐      ┌──────────────┐
     │ 1. Mine new  │      │ 2. Prune and │      │ 3. Check     │
     │    sessions  │      │    forget    │      │    health    │
     └──────┬──────┘      └──────┬───────┘      └──────┬───────┘
            │                     │                      │
            ▼                     ▼                      ▼
     New memories          Overloaded anchors     Novelty rate,
     hot-assigned          trimmed, global        utilization skew
     to existing           cap enforced           evaluated
     anchors                                            │
                                                        ▼
                                              ┌─────────────────┐
                                              │ Rebuild needed?  │
                                              └────────┬────────┘
                                                  yes  │  no
                                         ┌─────────────┼──────┐
                                         ▼                     │
                                  ┌─────────────┐             │
                                  │ 4. Rebuild   │             │
                                  │    atlas     │             │
                                  └──────┬──────┘             │
                                         │                     │
                                         ▼                     ▼
                                  ┌─────────────┐     ┌──────────────┐
                                  │ 5. Label     │     │ 5. Label     │
                                  │    queries   │     │    queries   │
                                  └──────┬──────┘     └──────┬───────┘
                                         │                     │
                                         ▼                     ▼
                                  ┌──────────────────────────────────┐
                                  │ 6. Train selector (if gates met) │
                                  └──────────────────────────────────┘
```

**Fast path** (every night with new data): Steps 1-2, plus labeling. Takes minutes.

**Slow path** (only when health triggers): Full rebuild + retrain. Takes hours. The system falls back to cosine selector during rebuild and switches to the trained selector once training completes.

New memories are **immediately searchable** — hot-assigned to the nearest existing anchor without waiting for a rebuild. The atlas rebuild re-clusters the full corpus, matches new clusters to existing anchors (preserving identity), and updates all assignments in a single transaction.

## Setup

### Requirements

- Python 3.9+
- [uv](https://docs.astral.sh/uv/) — Python package manager
- [Ollama](https://ollama.com/) running locally (for memory extraction — `qwen2.5:7b`)

```bash
# Install uv (if not already)
curl -LsSf https://astral.sh/uv/install.sh | sh

# Install Ollama and pull the model
ollama pull qwen2.5:7b
```

### Install from source

```bash
git clone https://github.com/ebilal/PSA
cd PSA
uv sync                    # installs dev deps (pytest, ruff, chromadb, faiss)
uv sync --extra training   # also installs torch for selector training
```

## Quick Start

### 1. Mine your sessions

```bash
uv run psa mine ~/.claude/projects/ --mode convos
```

Reads your Claude Code conversation transcripts, sends each through Qwen to extract typed memories (failures, procedures, facts, tool usage), embeds them, and stores them in SQLite. Takes a few hours for a large session history.

### 2. Build the atlas

```bash
uv run psa atlas build
```

Clusters your memories into semantic anchor regions via spherical k-means. Each anchor gets a Qwen-generated description (name, meaning, include/exclude terms). Requires 200+ memories. This is what makes search work — queries are matched to anchors, not individual memories.

### 3. Search

```bash
uv run psa search "why did we switch to GraphQL"
```

Embeds the query, retrieves the top-24 anchor candidates (BM25 + dense), selects the 1-4 best anchors, fetches their memories, ranks by relevance, and packs into a 6000-token context organized by type (failures first, then procedures, tool notes, episodes, facts). Works immediately with cosine selector.

### 4. Run the lifecycle once (labels queries and trains the selector)

```bash
uv run psa lifecycle run
```

This runs the full pipeline end-to-end: mines any remaining sessions, labels all available queries for selector training, rebuilds the atlas if health triggers, and trains the selector if enough labels (300+) have accumulated. The first run takes several hours because it labels all queries from scratch.

### 5. Install the nightly lifecycle

```bash
uv run psa lifecycle install
```

Installs a macOS launchd job that runs at midnight nightly. Each run:

1. **Mines** new Claude Code sessions added since last run
2. **Prunes** overloaded anchors and enforces the 50k memory cap
3. **Rebuilds** the atlas when health metrics trigger (novelty rate > 8%, skew > 3x)
4. **Labels** remaining queries for selector training (use `--label-batch 30` to cap per night)
5. **Trains** the selector once 300+ oracle labels accumulate — fine-tunes a cross-encoder that learns task utility (not just similarity), then activates it automatically

You can run it manually anytime: `uv run psa lifecycle run`

### 6. Add the MCP server to Claude Code

```bash
claude mcp add psa -- uv run --project /path/to/PSA python -m psa.mcp_server
```

Claude gets PSA tools — `psa_atlas_search`, `psa_store_memory`, `psa_atlas_status`, and more. When you ask a question about past work, Claude calls `psa_atlas_search` automatically and gets typed context from your memory store.

## Commands

### Mining

```bash
psa mine <dir>                          # project files (code, docs, notes)
psa mine <dir> --mode convos            # conversation exports (Claude Code, ChatGPT, Slack)
psa mine <dir> --dry-run                # preview without storing
psa mine <dir> --limit 50              # stop after N files
```

### Search

```bash
psa search "query"                      # PSA pipeline search
psa search "query" --results 10         # top-N results
```

### Atlas

```bash
psa atlas build                         # build atlas from current memories (requires 200+)
psa atlas status                        # version, anchor count, memory count
psa atlas health                        # novelty rate, utilization skew, rebuild recommendation
psa atlas rebuild                       # force rebuild
```

### Labeling

```bash
psa label                               # label all available queries
psa label --n-queries 100               # label only 100 queries
psa label --reset                       # delete all labels and start from scratch
psa label --sessions-dir ~/chats        # use a custom sessions directory
```

Each label runs a query through the pipeline, has the LLM judge which anchor sets actually help answer it, and saves the result. You need 300+ labels before the selector can be trained. Uses the cloud LLM if configured, otherwise falls back to local Ollama.

### Training

```bash
psa train                               # train selector (requires 300+ labels)
psa train --force                       # train even with fewer labels
```

Generates training data from oracle labels and fine-tunes the cross-encoder selector. Once trained, the selector learns task utility — not just topical similarity — and activates automatically.

### Lifecycle

```bash
psa lifecycle run                       # run the full pipeline (mine, prune, rebuild, label, train)
psa lifecycle run --label-batch 30      # limit labeling to 30 queries per run (~90 min)
psa lifecycle status                    # show last run, memory count, selector mode
psa lifecycle install                   # install nightly cron (midnight, labels all remaining)
psa lifecycle install --label-batch 30  # install with a per-night labeling cap
psa lifecycle uninstall                 # remove nightly cron
```

### Other

```bash
psa status                              # memory store overview
psa wake-up                             # atlas summary for session loading
psa benchmark --query "query"           # compare PSA vs raw ChromaDB side-by-side
psa migrate                             # migrate legacy ChromaDB drawers → PSA
```

  "psa_mode": "primary",

|-----|---------|-------------|

### LLM Configuration

PSA needs an LLM for memory extraction, anchor card generation, and oracle labeling. It tries a cloud API first (via [litellm](https://docs.litellm.ai/)), then falls back to local Ollama.

Configure in `~/.psa/llm.json` (created automatically on first run):

```json
{
  "provider": "cloud",
  "cloud_model": "azure/grok-4-20-reasoning",
  "cloud_api_key": "your-api-key-here",
  "cloud_api_base": "https://your-endpoint.openai.azure.com/openai/v1/",
  "local_endpoint": "http://localhost:11434/v1/chat/completions",
  "local_model": "qwen2.5:7b"
}
```

| Key | Description |
|-----|-------------|
| `provider` | `"cloud"` = try cloud first, fall back to local. `"local"` = local Ollama only. |
| `cloud_model` | Any litellm model string: `"azure/gpt-4o"`, `"openai/gpt-4o"`, `"anthropic/claude-sonnet-4-20250514"`, etc. |
| `cloud_api_key` | API key for the cloud provider |
| `cloud_api_base` | Base URL (required for Azure, optional for OpenAI/Anthropic) |
| `local_endpoint` | Ollama endpoint URL |
| `local_model` | Local model name (e.g., `"qwen2.5:7b"`, `"llama3:8b"`) |

To use only local Ollama (no cloud):
```json
{
  "provider": "local"
}
```

Environment variables override the config file: `PSA_LLM_MODEL`, `PSA_LLM_API_KEY`, `PSA_LLM_API_BASE`, `QWEN_ENDPOINT`, `QWEN_MODEL`.

## Lifecycle Management

PSA includes a lifecycle system that handles continuous learning, memory forgetting, and selector maintenance automatically.

### How It Works

New memories are **immediately searchable** — when you mine new sessions, each memory is hot-assigned to the nearest existing anchor without waiting for an atlas rebuild.

PSA tracks usage per memory:
- **select_count** — how many times a memory was fetched for a selected anchor
- **pack_count** — how many times it made it into the final packed context

Memories that are never used gradually become candidates for archival and eventual deletion.

### Nightly Pipeline

The lifecycle pipeline runs in two speeds:

**Fast path (every night with new data):** mine new sessions, hot-assign to existing anchors, prune overloaded anchors, enforce global memory cap. Takes minutes.

**Slow path (only when health triggers):** rebuild atlas (with anchor identity matching), label queries, retrain selector. Takes hours. During rebuild, the system falls back to cosine selector automatically.

### Forgetting

Memories are scored for disposability based on four factors:
- **Idle time** — how long since last packed into context
- **Anchor crowding** — whether the anchor exceeds its budget (default 100 memories)
- **Usage** — frequently packed memories are protected
- **Quality** — high-quality memories are protected

When an anchor exceeds its budget, lowest-scoring memories are archived (soft-deleted). Archived memories are hard-deleted after 90 days. A global cap of 50,000 active memories prevents unbounded growth. Newly mined memories have a 24-hour grace period before they can be pruned.

### Anchor Persistence

Anchors are durable semantic slots, not disposable k-means buckets. When the atlas rebuilds:
- New clusters are matched to existing anchors by centroid similarity
- Matched anchors keep their ID and stable card text (name, meaning, terms)
- Only genuinely novel clusters get new IDs
- Old anchors with no match are retired

The selector trains and infers on **stable card text only** (name + meaning + terms, no volatile examples). This preserves selector validity across atlas rebuilds.

## MCP Server — Claude CLI Integration

PSA integrates with Claude Code as a memory server via MCP. When Claude calls
`psa_atlas_search`, it receives a formatted context block drawn from your typed
memory objects — identical to what `psa inspect` shows you.

### Install

```
claude mcp add psa -- python -m psa.mcp_server
```

This registers PSA as a persistent MCP server. Claude calls `psa_status` on
session start, which returns the PALACE_PROTOCOL instructing Claude to call
`psa_atlas_search` before answering about any past decision or event.

### What Claude receives

`psa_atlas_search` runs the full pipeline and returns formatted context like:

```
PSA MEMORY CONTEXT — what auth pattern did we use?
============================================================

FAILURE WARNINGS
- Auth token stored in localStorage — caused XSS exposure in March

PROCEDURAL GUIDANCE
- Always validate JWT expiry server-side before trusting claims

FACTS & CONCEPTS
- RS256 signed JWTs with 15-minute expiry and refresh tokens
```

### Available MCP Tools

| Tool | Purpose |
|------|---------|
| `psa_atlas_search` | Full pipeline query — retrieves and packs relevant memories |
| `psa_store_memory` | Store a typed memory object directly |
| `psa_status` | Palace overview + wake-up protocol |
| `psa_atlas_status` | Atlas version, anchor count, memory count |
| `psa_atlas_health` | Health report (novelty rate, utilization skew) |
| `psa_list_anchors` | All anchors with utilization stats |
| `psa_rebuild_atlas` | Trigger atlas rebuild |
| `psa_search` | Legacy ChromaDB semantic search |
| `psa_add_drawer` | Store raw content in the legacy palace |

## Auto-Save Hooks (Claude Code)

Two hooks that automatically save memories during work sessions:

```json
{
  "hooks": {
    "Stop": [{"matcher": "", "hooks": [{"type": "command", "command": "/path/to/hooks/mempal_save_hook.sh"}]}],
    "PreCompact": [{"matcher": "", "hooks": [{"type": "command", "command": "/path/to/hooks/mempal_precompact_hook.sh"}]}]
  }
}
```

- **Save hook** — fires every 15 messages, prompts the AI to save key context
- **PreCompact hook** — emergency save before context compression

Configure in `.claude/settings.local.json`.
