# PSA -- Persistent Semantic Atlas

A memory system for AI coding agents. PSA ingests your conversation history, extracts typed memories, and retrieves them using learned models -- not flat embedding search.

Every conversation you have with an AI disappears when the session ends. PSA changes that. It mines your past sessions, builds typed memory objects (facts, procedures, failures, tool quirks), clusters them into a semantic atlas, and serves relevant context back to your AI agent. Over time, it learns which memories actually help answer your questions and gets better at finding them.

Works with Claude Code, Codex, and any AI tool that supports MCP or session hooks.

## Install

### Requirements

- Python 3.9+ (3.13 recommended)
- [uv](https://docs.astral.sh/uv/) package manager
- [Ollama](https://ollama.com/) running locally with `qwen2.5:7b`

### Setup

```bash
# Install uv
curl -LsSf https://astral.sh/uv/install.sh | sh

# Install Ollama and pull the model
brew install ollama
ollama pull qwen2.5:7b

# Clone and install PSA
git clone https://github.com/ebilal/memnexus.git
cd memnexus
uv sync                       # core + dev dependencies
uv sync --extra training      # also installs PyTorch for model training
```

### LLM Configuration

PSA uses an LLM for memory extraction and oracle labeling. By default it uses local Ollama. To use a cloud API (faster, better quality), edit `~/.psa/llm.json`:

```json
{
  "provider": "cloud",
  "cloud_model": "gpt-4o-mini",
  "cloud_api_key": "sk-...",
  "local_endpoint": "http://localhost:11434/v1/chat/completions",
  "local_model": "qwen2.5:7b",
  "local_fallback": true
}
```

Set `"provider": "local"` to use only Ollama (no API key needed).

## Quick Start

### 1. Mine your conversations

```bash
psa mine ~/.claude/projects/ --mode convos
```

Reads Claude Code session transcripts, extracts typed memories via Qwen, embeds them, stores in SQLite. Takes a few minutes to hours depending on history size.

### 2. Build the atlas

```bash
psa atlas build
```

Clusters memories into 256 semantic anchor regions via spherical k-means. Each anchor gets a description generated by the LLM. Requires 200+ memories.

### 3. Search

```bash
psa search "why did we switch to GraphQL"
```

Returns packed context organized by type -- failures first, then procedures, tool notes, facts.

### 4. Train the models (optional, improves quality)

```bash
psa label --n-queries 300        # label queries for training (~1-2 hours)
psa train --force --coactivation # train selector + co-activation model
```

### 5. Connect to Claude Code

```bash
claude mcp add psa -- uv run --project /path/to/memnexus python -m psa.mcp_server
```

Claude gets `psa_atlas_search` and other memory tools. It automatically retrieves relevant context when you ask about past work.

### 6. Install nightly lifecycle

```bash
psa lifecycle install
```

Runs at midnight: mines new sessions, prunes old memories, rebuilds atlas when needed, retrains models.

## How It Works

### The Philosophy

Memory is not data retrieval. You don't look up what you know -- you just know it. PSA is built around this idea. Instead of searching through a flat vector database, PSA learns a two-level activation model that "knows" which regions of memory space are relevant to a query, then scores individual memories within those regions.

### Two Pipelines

**Ingest:** Raw text (conversations, code, docs) goes through structure-aware chunking, LLM extraction into six typed memory objects, quality filtering, deduplication, and embedding. Raw sources are stored immutably; memory objects are derived indexes over them.

**Query:** A two-level learned model activates the right memory regions and selects the best individual memories:

```
Query
  |
  v
Level 1: Full-Atlas Activation
  - Cross-encoder scores ALL 256 anchors (not just top-24)
  - Co-activation transformer learns inter-anchor relationships
  - Adaptive threshold selects variable number of anchors (1-20)
  - Enriched with per-anchor type distribution + quality features
  |
  v
Level 2: Memory-Level Scoring
  - Cross-encoder scores (query, memory_body) for each memory
  - MLP re-ranker combines textual relevance + type + quality + recency
  - Outputs ranked memories; packer fills token budget in this order
  |
  v
Packed Context (6000 tokens, type-organized)
```

### Six Memory Types

| Type | What it captures | Priority |
|------|-----------------|----------|
| FAILURE | What went wrong and why | Highest |
| PROCEDURAL | How-to steps, workflows | High |
| TOOL_USE | Tool/API behavior, flags, quirks | High |
| EPISODIC | Specific events, debugging sessions | Medium |
| SEMANTIC | Facts, concepts, domain knowledge | Medium |
| WORKING_DERIVATIVE | Scratch notes, intermediate reasoning | Low |

Failures are packed first because they prevent repeating mistakes.

### The Atlas

The atlas maps your memory space into 256 semantic regions (224 learned + 32 novelty). Instead of searching every memory individually, PSA matches your query to a handful of anchor regions, then digs into the memories behind them.

Each anchor has:
- A centroid (768-dim embedding, cluster center)
- A card (name, meaning, include/exclude terms, query patterns)
- Type distribution (what kinds of memories it contains)
- Quality and count metadata

Anchors are durable -- when the atlas rebuilds, new clusters are matched to existing anchors by centroid similarity. Only genuinely novel clusters get new IDs. The trained models remain valid across rebuilds.

### Learning Loop

PSA improves over time through a nightly lifecycle:

1. **Mine** new sessions added since last run
2. **Prune** overloaded anchors, enforce memory cap (50k)
3. **Check health** -- novelty rate > 8% or skew > 3x triggers rebuild
4. **Rebuild atlas** if needed (preserves anchor identity)
5. **Label queries** -- LLM judges which anchor sets help answer real queries
6. **Train models** -- cross-encoder selector, co-activation transformer, memory re-ranker

New memories are immediately searchable -- hot-assigned to the nearest anchor without waiting for a rebuild.

### Forgetting

PSA doesn't grow unboundedly. Memories are scored for disposability based on:
- Idle time (days since last used)
- Anchor crowding (too many memories in one anchor)
- Usage (frequently packed memories are protected)
- Quality (high-quality memories are protected)

Low-scoring memories are archived first (soft delete), then hard-deleted after 90 days. Newly mined memories get a 24-hour grace period.

## Commands

### Mining

```bash
psa mine <dir>                    # mine project files (code, docs, notes)
psa mine <dir> --mode convos      # mine conversation exports
psa mine <dir> --dry-run          # preview without storing
psa mine <dir> --limit 50         # stop after N files
```

Supported conversation formats: Claude Code JSONL, Claude AI JSON, ChatGPT exports, Codex JSONL, Slack JSON, plain text.

### Search

```bash
psa search "query"                # full pipeline search
psa search "query" --results 10   # top-N results
```

### Atlas

```bash
psa atlas build                   # build atlas (requires 200+ memories)
psa atlas status                  # version, anchor count, memory count
psa atlas health                  # novelty rate, skew, rebuild recommendation
psa atlas rebuild                 # force rebuild
```

Use `--tenant <name>` to operate on a specific tenant (default: "default").

### Labeling

```bash
psa label                         # label queries for selector training
psa label --n-queries 100         # label only N queries
psa label --reset                 # delete all labels, start fresh
```

Each label runs a query through the pipeline and has the LLM judge which anchor sets help answer it. You need 300+ labels before the selector trains.

### Training

```bash
psa train                         # train cross-encoder selector
psa train --force                 # train even if gates not met
psa train --coactivation          # also train co-activation model
psa train --memory-scorer         # also train Level 2 memory re-ranker
psa train --force --coactivation --memory-scorer  # train everything
```

The selector uses a three-phase curriculum: warm start on easy examples, hard-negative discrimination, adversarial query rewrites. The co-activation model trains a 2-layer transformer on anchor co-occurrence patterns. The memory scorer trains a tiny MLP on (query, memory) relevance.

### Lifecycle

```bash
psa lifecycle run                 # run full pipeline
psa lifecycle run --label-batch 30  # cap labeling at 30 queries
psa lifecycle status              # last run, memory count, selector mode
psa lifecycle install             # install nightly cron
psa lifecycle uninstall           # remove nightly cron
```

### Benchmark

```bash
psa benchmark longmemeval ingest            # download + ingest LongMemEval dataset
psa benchmark longmemeval run               # run queries (default: cosine selector)
psa benchmark longmemeval run --selector coactivation  # run with full activation system
psa benchmark longmemeval score --results <path>       # score answers
psa benchmark longmemeval oracle-label --results <path> --mode fast  # generate oracle labels
```

### Inspection

```bash
psa inspect "query"               # see exactly what context PSA produces
psa inspect "query" --verbose     # show all 24 candidates with scores
psa log list                      # recent queries
psa log show <run_id>             # full entry as JSON
psa status                        # memory store overview
```

### Other

```bash
psa migrate                       # migrate legacy ChromaDB palace to PSA
psa compress                      # re-consolidate memories
psa split                         # split oversized source files
psa hook <harness> <event>        # session hooks (claude-code, codex)
```

## MCP Server

PSA integrates with Claude Code (and any MCP-compatible client) as a memory server.

### Install

```bash
claude mcp add psa -- uv run --project /path/to/memnexus python -m psa.mcp_server
```

### Available Tools

| Tool | Purpose |
|------|---------|
| `psa_atlas_search` | Full pipeline query -- retrieves and packs relevant memories |
| `psa_store_memory` | Store a typed memory object directly |
| `psa_status` | Memory store overview + protocol instructions |
| `psa_atlas_status` | Atlas version, anchor count |
| `psa_atlas_health` | Health report (novelty rate, skew) |
| `psa_list_anchors` | All anchors with utilization stats |
| `psa_rebuild_atlas` | Trigger atlas rebuild |
| `psa_search` | Legacy semantic search |
| `psa_check_duplicate` | Check if a memory already exists |

When Claude starts a session, it calls `psa_status` which returns instructions to use `psa_atlas_search` for any question about past work.

## Configuration

### Main Config (`~/.psa/config.json`)

| Key | Default | Description |
|-----|---------|-------------|
| `psa_mode` | `"primary"` | `"primary"`, `"side-by-side"`, or `"off"` |
| `tenant_id` | `"default"` | Active tenant (env: `PSA_TENANT_ID`) |
| `token_budget` | `6000` | Max tokens in packed context |
| `max_memories` | `50000` | Global memory cap |
| `anchor_memory_budget` | `100` | Per-anchor memory limit |

### LLM Config (`~/.psa/llm.json`)

| Key | Description |
|-----|-------------|
| `provider` | `"cloud"` (try cloud, fall back to local) or `"local"` (Ollama only) |
| `cloud_model` | Any litellm model string: `"gpt-4o-mini"`, `"azure/gpt-4o"`, etc. |
| `cloud_api_key` | API key for cloud provider |
| `local_model` | Local model name (default: `"qwen2.5:7b"`) |

Environment overrides: `PSA_LLM_MODEL`, `PSA_LLM_API_KEY`, `PSA_LLM_API_BASE`, `QWEN_ENDPOINT`.

## Benchmark Results

Evaluated on [LongMemEval](https://huggingface.co/datasets/xiaowu0162/longmemeval) (500 questions across long conversation histories):

| Configuration | R@5 | Exact F1 | LLM-as-Judge | Anchors/Query |
|--------------|-----|----------|--------------|---------------|
| Cosine baseline (k=6) | 0.856 | 0.202 | 0.330 | 6 (fixed) |
| Trained cross-encoder (rerank, k=6) | 0.894 | 0.172 | 0.352 | 6 (fixed) |
| Full-atlas co-activation v1 | 0.912 | 0.172 | 0.362 | 1-19 (adaptive) |
| **Two-level activation (current)** | **0.920** | **0.177** | **0.358** | 1-20 (adaptive) |

**R@5** = fraction of gold-standard anchors found in the selected set. The two-level system finds 92% of relevant memory regions, up from 85.6% with the cosine baseline.

**What each level contributes:**
- **Full-atlas scoring** eliminates the retriever bottleneck (26% of gold anchors were lost at the top-24 retrieval stage)
- **Co-activation transformer** learns inter-anchor relationships and selects a variable number of anchors per query
- **Memory-level scorer** re-ranks individual memories within selected anchors by learned relevance

All models run on a laptop (Apple Silicon MPS). Full query pipeline: ~340ms.
